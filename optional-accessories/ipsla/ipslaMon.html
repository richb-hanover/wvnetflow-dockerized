Usage:

ipslaMon.cgi [OPTION] [COMMAND]...

 

Options

 --datadir=DIRECTORY    Absolute path of data directory (chmod 664). Used for rrd, log, and config files.

 --graphdir=DIRECTORY   Absolute path of graph work directory (chmod 666). Should be underneath web server's DocumentRoot.

 --graphurl=RELATIVE    Relative path to graph work directory, for composing URIs.

Commands

 --poll                 Collect IP SLA data once. Use this from a scheduler like cron. The cron frequency should match the 'rrd period' config setting.

# crontab entry for ipslaMon with 'rrd period 60'

* * * * * /var/www/cgi-bin/ipslaMon.cgi --poll

 --daemon               Collect IP SLA data continuously. The collection interval is determined by 'rrd period' config setting.

 --setup                Attempt to set up ipslaMon.cgi directories with proper permissions and create a default config file.

Description:

ipslaMon.cgi is a script used for both backend collection of IP Service Level Agreement (IP SLA, formerly RTR or SAA) data from Cisco routers and for frontend viewing of this data from a web browser. IP SLA is a router technology that measures the response time of probe traffic across arbitrary network paths. It is good to have a basic idea of how to configure IP SLA probes before implementing this script. Refer to http://www.cisco.com/go/ipsla for more information.

Basic script installation steps:

1.    Begin with a machine with a web server and all dependencies (see below).

2.    Copy ipslaMon.cgi to a cgi directory (e.g., /var/www/cgi-bin).

3.    Choose or create a user to run the backend collection.

4.    As root, run 'ipslaMon.cgi --user=<username> --setup'.

5.    Configure the user with a cron job to run 'ipslaMon.cgi -poll' every minute.

6.    Open the script in a browser (e.g., http://myserver/cgi-bin/ipslaMon.cgi)

7.    Use the web interface to configure a few IP SLA source routers.

The web interface is populated with information learned from the IP SLA souce routers. Here is a sample IP SLA probe:

  ip sla 50

   type jitter dest-ipaddr 10.110.248.98 num-packets 5

   tos 0x2E

   tag 0234-Providence, RI (Frame-relay)

   frequency 300

   request-data-size 100

Guidelines for configuring IP SLA for use with ipslaMon.cgi:

  • only echo and jitter probes are supported (at this time). Some echo-like probes, such as http and dns requests, may work.
  • udp jitter probes may collect one-way times, but these are only accurate if both devices involved in the probe have local NTP servers (e.g., GPS clocks). If NTP is run across the WAN link being probed by IP SLA, the one-way times will be blank and/or incorrect.
  • ip sla 'frequency' should be >= the 'rrd period' of ipslaMon.cgi or else some data samples will be missed.
  • the ip sla index is persistent. If an index number is reused for a different target, the new tag will be used, but the graph will contain old data.
  • 'tag' is used for labeling and grouping sites within the web interface. Each tag should be made of a general target name followed by specific probe attributes in parenthesis. For example,

ip sla 50

 tag Madison, WI (via Frame Relay)

 

ip sla 51

 tag Madison, WI (via VPN Tunnel)

 

The web interface will show a menu with 'Madison, WI', from which a graph will show 'via Frame Relay' and 'via VPN Tunnel' as two different colors.

By creating multiple probes with different attributes and tags, it's easy to group the probe results on the same graph (e.g., for different network paths, QoS settings, packet sizes, etc).

Variable reference:

 

Below is a reference guide to all the variables configurable from the web interface's setup option.

 

router

Add each IP SLA source router to be monitored. The name you provide will be displayed in the graphs and tables.

 

Each router has two sub-variables:

snmp  The SNMP community string to use. Leave blank to use the global.

host  The name or IP address of the router. Leave blank to use the router name.

 

 

 

directory

 

 

data

The directory where rrd, config, cache, and log files are stored. The default is /var/ipsla.

 

graph_absolute

The directory where graphs are created and served up to the web user. This must be a web-accessible directory (i.e., under ‘DocumentRoot’) and be writeable by the web user (e.g., ‘nobody’ or ‘apache’). The default is /var/www/html/graphs.

 

graph_relative

The relative path that refers to the graph_absolute directory from the perspective of a web user. This is used to compose links to images.

 

warnings

By default, if ipslaMon.cgi thinks there is a problem with graph_absolute and graph_relative, it will display an error message. This may be wrong if your web server uses virtual directories or other such advanced tricks. Set this to ‘off’ to suppress these errors.

 

snmp

 

 

community

Global community string, which may be overridden on a per-router basis.

 

version

Version to use, ‘1’ or ‘2c’. The default is ‘2c’ which is far more efficient for bulk snmpwalks. Revert to version ‘1’ only if ‘2c’ isn’t working for you.

 

timeout

Timeout in seconds. Keep this low.

 

retries

Retry attempts.

 

backoff

Backoff time in seconds. Each successive retry waits this many more seconds.

 

port

UDP port to use.

 

 

rrd

This section defines how RRD files are created. It may be useful to refer to the rrdtool documentation at http://people.ee.ethz.ch/~oetiker/webtools/rrdtool/

 

Note: These parameters are used to initialize the RRD database files. If you change one or more parameters, it's important to reset all RRD files by using the reset link on the Admin page. This will delete all data history!

 

 

 

period

The RRD period indicates how frequently IP SLA samples will be collected, in seconds. Common values are 60 (1 minute), 300 (5 minutes), and 900 (15 minutes).

 

If a scheduler is used to run ipslaMon.cgi, be sure the execution frequency matches the rrd period.

 

It is okay for an IP SLA probe to have a frequency higher than the RRD period. However, if an IP SLA probe has a lower frequency, then samples will be lost.

 

rows_period

rows_hour

rows_day

Determines how many samples are kept for each resolution (period, hour, and day).

 

As data is collected at ‘period’ resolution, it is aggregated into hourly and daily values. This allows the RRD file to contain a year or more of samples without consuming a huge amount of disk space.

 

The default is to keep 380 rows of ‘day’ (about a year), 720 rows of ‘hour’ (30 days), and 2016 rows of ‘period’ (7 days if period is 300, or 1.4 days if period is 60).

 

Those familiar with RRD may notice that there is no way to specify 'steps', since steps are calculated automatically based on the period.

 

fudge

Determines the minimum number of seconds that must pass between sample updates. A good value is ‘10’. Fudge must be less than period.

 

heartbeat

Determines defines the maximum number of seconds that may pass between samples. Typically, samples are collected like clockwork every ‘rrd period’ seconds. Samples may be delayed or missing if there is a large number of routers, network congestion, or down routers. To see evidence of lost samples, set the heartbeat to be 20-30 seconds over the ‘rrd period’. If you prefer nicer looking graphs without missing data, set heartbeat to 2-3 times the value of period. The default is 930.

 

xff

When samples are consolidated into hourly and daily values, there may be some missing values (see heartbeat). The XFF "defines what part of a consolidation interval may be made up from (missing) data while the consolidated value is still regarded as known." (RRD manual)

 

 

graph

 

 

default_range

The default setting for the range pull-down, measured in seconds. The default value is 32400 (9 hours).

 

format

The file format for graph files, ‘png’ or ‘gif’. The default is ‘png’.

 

huge_height

huge_width

large_height

large_width

medium_height

medium_width

small_height

small_width

tiny_height

tiny_width

The height and width (in pixels) of the various graph sizes.

 

jitter_combined

jitter_separate

There is a tremendous amount data collected from IP SLA jitter probes. By default, all of it is displayed. It may be useful to reduce the number of graphs by changing these settings.

 

‘jitter_combined’ determines whether or not a graph will be shown of the combined positive and negative jitter measurements. The default is ‘yes’.

 

‘jitter_separate’ determines whether or not a graph will be shown of the individual positive and negative jitter measurements. The default is ‘yes’.

 

misc

 

 

cachefile

The filename for the cachefile (ipsla.cache). This is stored in the data directory.

 

logfile

The filename for the log file (ipsla.log). This is stored in the data directory.

 

linkname

linkurl

Defines a navigation link that should be displayed on every web page.

 

‘linkname’ contains visible link name. Use ‘%20’ instead of space characters (e.g., ‘Our%20Home%20Page’).

 

‘linkurl’ contains the URL.

 

loglevel

Determines how much log data to display.

   1 only error messages

   2 error and informational messages

   3 debug messages

 

loglines

Determines how many lines of log data should be shown on the ‘IP SLA Log’ web link.

 

logsize

Determines how much log data should be kept, period. When the log file reaches this size, it is moved to ipsla.log.bak. Thus, the actual disk space may be twice the specified logsize.

 

password

Sets a assword that controls access to the ipslaMon.cgi Setup screen.

 

tagregexp

Specifies the regular expression that will split an IP SLA probe’s tag into a descriptive name and probe-specific information. The name must be left in the first match argument $1.

 

The default is ^([^\()*)

 

timer

 

 

cache

Controls how often the IP SLA config data is reloaded from the routers, in seconds. The default is 3600 (one hour).

 

A config change to a router may have to wait this much time before the IP SLA probe begins to be tracked.

 

cleanup

Controls how frequently graph files are purged. The default is 600 (20 minutes).

 

color

Colors may be specified as ‘#RRGGBB’ or as a common color name such as ‘red’, ‘green’, or ‘fuscia’.

 

 

 

background

Specifies the background color for each graph.

 

baseline

Specifies the color for the baseline (zero).

 

1

2

3

etc…

Specifies the color for each line element. Any number of lines can be defined. If a graph contains more elements than defined line colors, colors are reused.
